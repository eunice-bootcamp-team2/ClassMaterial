	WSL은 “윈도우 안에 작은 리눅스 컴퓨터를 넣는 것”이에요.  
	→ 그래서 배포할 때랑 똑같은 환경에서 미리 연습하려고 설치합니다.

❓ 왜 굳이 그렇게 할까?

1️⃣ **실제 서버는 리눅스에서 돌아가요**

우리가 나중에 올릴 곳들:
- AWS / EC2
- 클라우드 서버
- Docker, Kubernetes
- 배포된 Django / FastAPI
    
👉 다 리눅스에서 실행됩니다.
> 로컬에서 윈도우로 개발 → 서버에 올리면  
> 여기선 되는데, 서버에서만 안 돼요 😭 가 자주 생김

WSL을 쓰면:
> **로컬 = 서버 환경이랑 거의 동일**  
> ➡️ 배포할 때 문제 훨씬 줄어듭니다

---
2️⃣ 패키지 설치가 리눅스에서 훨씬 안정적
	파이썬, AI, 백엔드 라이브러리 중 상당수가 리눅스 기준으로 만들어져 있어요.

- 윈도우 → 오류, 꼼수 필요 ❌
- 리눅스 → 공식 방법 그대로 ⭕
    

WSL = 리눅스니까 그냥:
`pip install ...`
그대로 잘 됩니다.

---
3️⃣ “연습할 때부터 서버처럼” 해보기
	WSL을 쓰면 VSCode가 이렇게 됩니다:

> 화면 : 윈도우  
> 실행 : 리눅스(서버와 동일)

👉 실제 서버 다루는 연습을, 지금부터 자연스럽게 하는 셈이에요.

---
4️⃣ 윈도우를 지우지 않아도 됨 (이게 중요!)

리눅스가 필요하지만:
> “윈도우를 포맷하고 리눅스를 깔자” ❌ 너무 위험하고 번거롭죠.

그래서:
> **윈도우 안에 리눅스를 “살짝” 설치 → WSL**

👉 안전, 빠름, 편함.

---
초보자용 비유로 이해해볼까요?

🏠 **윈도우 = 우리 집**  
🐧 **리눅스 = 회사(서버)에서 쓰는 시스템**

WSL은 이렇게 말하는 거랑 같아요:
> “회사 시스템을, 집 안에 작은 방 하나 만들어서 똑같이 설치해 두자.”

그러면:
- 집에서 연습해 보고
- 회사 가면 → 똑같이 써서 실수 줄이기 👍
    
---

###### ✅ 정리하면
| 이유          | 설명               |
| ----------- | ---------------- |
| 서버와 같게 만들려고 | 배포할 때 에러 줄이기     |
| 라이브러리 호환    | 리눅스가 훨씬 안정적      |
| 학습 연속성      | 로컬 + 서버 방식 동일    |
| 안전          | 윈도우는 유지, 리눅스만 추가 |

👉 그래서 **WSL을 설치한 거예요.**

---
❌ 리눅스를 깊게 공부할 필요 없음  
⭕ **CLI 명령어 + 개발에 필요한 것만** 익히면 충분

지금처럼:
- VSCode 열고
- 터미널에서 명령 조금씩 써 보고
- 프로젝트 하면서 익히면
    
👉 자연스럽게 “배포할 줄 아는 개발자”로 연결됩니다 😊

---
왜 맥(Mac)에서는 WSL을 설치할 필요가 없을까?

결론부터:
> 맥은 원래 리눅스와 거의 같은 구조라서,  
> WSL처럼 “리눅스를 따로 깔 필요가 없습니다.”

1️⃣ 맥OS = “유닉스 기반”
	맥은 기본 구조가 이렇게 생겼어요:
```
Mac  ≈  Unix  (리눅스와 같은 계열)
```

그래서:
- 명령어 방식이 리눅스와 거의 같고
- 서버 환경과 차이가 거의 없어요
- 터미널만 열면 바로 리눅스처럼 사용 가능
    
👉 이미 “개발자용 환경”이 내장되어 있는 셈입니다.

2️⃣ 윈도우는 구조가 달라서 WSL이 필요
윈도우는 이렇게 생겼어요:
```
Windows  ≠  Linux
```

그래서:
- 서버랑 환경이 많이 다르고
- 리눅스용 도구들이 잘 안 돌아가서  
    ➡️ **리눅스를 하나 더 넣어주는 도구 = WSL**이 필요했던 것.

|OS|리눅스처럼 쓰려면?|이유|
|---|---|---|
|**Mac**|❌ 필요 없음|처음부터 유닉스(리눅스와 친척)|
|**Windows**|✔️ WSL 필요|구조가 완전히 달라서|

맥은 VSCode만 설치하면 “터미널(리눅스처럼 쓰는 환경)”이 이미 있어요.  
그래서 WSL 같은 추가 설정이 필요하지 않습니다.
```
Mac 터미널
   ↓
zsh (예전에는 bash)
   ↓
리눅스랑 거의 같은 명령어 사용
```

bash랑 zsh는 **거의 동일하게 쓰입니다.**  
(명령어 95% 동일 — 배우는 데 차이 없음 👍)

그래서 맥에서는:

- 터미널 열기
- VSCode 열기
- 바로 명령어 사용 가능
    
➡️ 별도의 리눅스 설치(Wsl) 필요 없음

맥은 개발용 리눅스 환경이 이미 내장되어 있어서, VSCode만 깔면 바로 CLI/서버 연습이 가능하다.

📁 파일/폴더 다루기
```bash
ls            # 현재 폴더의 파일/폴더 목록 보기
ls -l         # 자세히 보기(권한, 날짜, 크기 등)

pwd           # 내가 지금 있는 위치 확인

cd projects   # projects 폴더로 이동
cd ..         # 한 단계 위로 이동
cd ~          # 홈 디렉토리로 이동

mkdir test    # test 라는 폴더 만들기
rmdir test    # (비어 있는) test 폴더 삭제
```

📄 파일 만들고 / 이동 / 삭제
```bash
touch memo.txt           # 빈 파일 생성
rm memo.txt              # 파일 삭제
rm -r myfolder           # 폴더 + 안의 내용 삭제(조심!)

cp a.txt b.txt           # a.txt 를 b.txt 로 복사
mv a.txt folder/         # 파일을 다른 폴더로 이동
mv old.txt new.txt       # 파일 이름 변경
```

🔍 파일 내용 보기
```bash
cat memo.txt             # 파일 전체 내용 보기
head memo.txt            # 위쪽 몇 줄만 보기
tail memo.txt            # 아래쪽 몇 줄만 보기
tail -f log.txt          # 로그 파일이 추가되는 것을 실시간으로 보기
```

⚙️ 권한 / 관리자 권한
```bash
chmod +x script.sh       # 실행 권한 주기
sudo apt update          # 관리자 권한으로 패키지 목록 업데이트
sudo apt install git     # git 설치
```
	sudo = 관리자 권한으로 실행 (Windows의 “관리자 실행” 느낌)

🐍 파이썬 관련 자주 쓰는 것
```bash
python3 main.py          # 파이썬 실행
pip install requests     # 파이썬 패키지 설치
```

---
**`python3 파일.py` 와 `python 파일.py` 의 차이**는 어떤 파이썬 버전을 실행하느냐 입니다.

🐍 기본 개념
	예전에는 파이썬이 이렇게 나뉘어 있었어요:
- Python 2 (옛날 버전)
- Python 3 (지금 표준)
    
그래서 리눅스/맥에서는 보통 이렇게 구분했습니다:
```
python   → Python 2
python3  → Python 3
```
➜ 그래서 확실히 Python 3를 쓰고 싶을 때 `python3 파일.py` 라고 실행한 거예요.

그런데 지금은 상황이 조금 달라짐
요즘 대부분 환경에서는:
```
python   → Python 3
python3  → 역시 Python 3 (같은 걸 가리키기도 함)
```
	즉, 둘 다 같은 프로그램을 실행하는 경우도 많아요.

확인해보세요 👇
터미널에서:
```
python --version
python3 --version
```
결과가 같으면 → 둘 다 Python 3 😊  
다르면 → `python3`를 사용하세요.

환경별 요약

🐧 WSL / 리눅스 / 맥

보통 이렇게 생각하면 편해요:

|명령|의미|
|---|---|
|`python3`|Python 3 (권장)|
|`python`|Python 2 일 수도 있음 (환경 따라 다름)|

✔️ 가장 안전한 기본 규칙:

> **리눅스/WSL/맥 → `python3`**  
> **윈도우 → `python` 또는 `py -3`**

✔️ 헷갈리면 항상 먼저 버전 확인:
```bash
python --version 
python3 --version
```
